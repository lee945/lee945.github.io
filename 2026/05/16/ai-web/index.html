4.4 为什么需要反弹 Shell

sqlmap 的 os-shell 本质是 HTTP 请求-响应模式，每次命令都是独立的 HTTP 请求，不支持交互式程序（sudo、su、vi 等）。反弹 shell 建立的是持续 TCP 连接，可以正常进行交互操作，是后续提权的前提。

6.6 其他提权思路（备忘）

/etc/passwd 可写只是其中一种提权方式。常规提权检查清单：

  • sudo -l — 查看 sudo 权限
  • find / -perm -4000 -type f 2>/dev/null — 查找 SUID 文件
  • cat /etc/crontab — 查看计划任务
  • uname -a — 检查内核版本（Dirty Cow、Dirty Pipe 等内核漏洞）
  • find / -writable -type f 2>/dev/null | grep -v /proc — 查找全局可写文件

八、总结

攻击链回顾

关键漏洞点

漏洞 说明 危害
信息泄露 robots.txt + phpinfo 暴露敏感路径 为 getshell 提供路径信息
SQL 注入 uid 参数未过滤，双引号闭合 数据库泄露 + 写 webshell
目录权限不当 uploads 目录对 MySQL 进程可写 允许写入 PHP 并执行
/etc/passwd 可写 文件权限配置不当 直接添加 root 用户提权

反思

  1. 信息收集是关键 — phpinfo 泄露的绝对路径是 getshell 的核心前提，robots.txt 常常被忽略但往往直接暴露敏感路径
  2. 先评估权限再动手 — 提前检查 DBA/FILE/secure_file_priv 可以避免无效尝试，帮助确定技术路线
  3. 目录权限比根目录重要 — 根目录不可写不代表 uploads/temp/cache 不可写，逐层尝试
  4. 反弹 shell 是提权的前提 — os-shell 无法交互（su 需要终端），稳定 shell 后才能进行提权操作
  5. 有 FILE 权限不代表能直接写文件 — secure_file_priv = NULL 时 INTO OUTFILE 被封，但 –file-write / –os-shell 等其他路径仍然可用
  6. 蚁剑提升效率 — 图形化管理文件、终端、数据库比命令行手动操作高效很多
