第一章：命令执行基础

1.1 概念与原理

命令执行漏洞是指服务端未对用户输入进行过滤，导致用户可以通过拼接系统命令，使服务器执行任意系统命令。

Web 程序、Web 框架和 Web 组件等有时需要调用系统命令来完成特定功能（如压缩、转换格式、发送邮件等）。如果这些调用拼接了用户可控的输入且未做过滤，就会产生命令执行漏洞。

1.2 漏洞利用条件

  1. 用户能够控制函数的输入
  2. 存在可执行系统命令的危险函数

1.3 漏洞产生原因

  1. 未对用户输入进行过滤或过滤不严：例如未过滤 &、&&、|、|| 等连接符
  2. 系统漏洞造成的命令执行：如 Bash 破壳漏洞（CVE-2014-6271），可构造环境变量的值执行恶意脚本代码，影响通过 Bash 交互的多种应用（HTTP、SSH、DHCP 等）
  3. 第三方组件存在代码执行漏洞：如 Struts2、Elasticsearch Groovy、ThinkPHP 等框架的历史漏洞

1.4 常见的危险函数

语言/平台 危险函数
PHP exec、shell_exec、system、passthru、popen、proc_open、eval、assert 等
ASP.NET System.Diagnostics.Process.Start、System.Diagnostics.ProcessStartInfo 等
Java java.lang.Runtime.getRuntime()、java.lang.Runtime.exec() 等

1.5 PHP 危险函数详解

以下以 PHP 5/7 环境（端口 80）为例，逐一演示各危险函数的使用方式。

system()

system(string $command, int &$return_var = ?)：执行 command 参数所指定的命令，直接输出执行结果。如果提供了 return_var 参数，则外部命令执行后的返回状态将会被设置到此变量中。

http://192.168.137.151:18080/class01/system.php?benben=ls

特点：直接回显，自动输出结果。

exec()

exec(string $command, array &$output = ?, int &$return_var = ?)：执行命令。单独使用时只返回最后一行结果且不回显。$output 参数会逐行被命令输出填充。

http://192.168.137.151:18080/class01/exec.php?cmd=ls

特点：无直接回显，需借助 print_r 打印 $output 数组才能看到结果。

passthru()

passthru(string $command, int &$return_var = ?)：执行命令，把原始输出直接送到浏览器/终端。常用于输出二进制流或原始命令结果。

http://192.168.137.151:18080/class01/passthru.php?cmd=ls

特点：类似 system() 但更底层，适合输出二进制数据。

shell_exec() 与反引号

shell_exec(string $cmd)：在 shell 环境中执行命令，将完整输出以字符串方式返回。功能等同于反引号。

特点：两者等价，反引号更简洁，都需借助 echo/print 输出结果。

popen()

popen(string $command, string $mode)：打开一个指向进程的文件指针。mode 参数：'r' 表示读取，'w' 表示写入。需通过 fgets 逐行读取输出。

http://192.168.137.151:18080/class01/popen.php?cmd=ls

特点：单向管道，只能读或写；需手动读取。

proc_open()

proc_open(string $command, array $descriptor_spec, array &$pipes, ...)：比 popen() 更强大的进程控制函数。$descriptor_spec 是一个定义 I/O 管道的数组，索引 0 为标准输入、1 为标准输出、2 为标准错误。

特点：支持双向通信，可同时控制 stdin/stdout/stderr，是功能最全的进程执行函数。

命令执行危险函数对比总结：

函数 回显方式 特点
system() 直接输出 最简单直接
exec() 需 print_r($array) 逐行存入数组，仅返回最后一行
passthru() 直接输出原始数据 适合二进制流
shell_exec() 需 echo 返回完整字符串
反引号 需 echo 等同于 shell_exec()
popen() 需 fgets 逐行读 单向管道
proc_open() 需 stream_get_contents 双向通信，最强大

第二章：命令执行 — 过滤绕过

2.1 连接符绕过

目标地址：http://192.168.137.151:18080/class03/1.php

源码分析：

命令被硬拼接在 ls 之后，所以 cmd=;id 实际执行的是 ls;id。以下是各种连接符的行为：

分号（;）— 顺序执行

分号按顺序依次执行每条命令，无论前一条成功与否。

管道符（|）— 输出传递

将前一条命令的 stdout 传递给后一条命令的 stdin。注：cmd1 | cmd2 只保留 cmd2 的输出。

或运算符（||）— 短路执行

前一条成功（返回 0）则后一条不执行；前一条失败则执行后一条。

变体场景：有些题目会将用户输入重定向到 /dev/null 导致无回显：

此时利用 || 确保重定向命令不执行：cmd=id||，实际执行 id|| >/dev/null 2>&1，id 成功后短路，不执行后面的重定向，从而获得回显。

连接符完整对比：

` ` 管道，stdout→stdin
` `

& 后台执行前一条 sleep 10&id
%0a / %0d%0a 换行符，相当于分号 ls%0aid

第三章：命令执行 — disable_functions 绕过

当 disable_functions 禁用了 system、exec、shell_exec、passthru、popen、proc_open 等所有可能执行系统命令的函数时，需要”曲线救国”。本章介绍两种绕过思路。

3.1 LD_PRELOAD 劫持（mail 函数）

目标地址：http://192.168.137.151:18081/class02/2.php

原理

Linux 动态链接器 ld.so 在加载程序时，会优先加载 LD_PRELOAD 环境变量指定的共享库（.so）。通过劫持 sendmail 必然会调用的 libc 函数，可以在 sendmail 进程的上下文中执行任意命令。

攻击链路：

绕过条件：

  1. 能够上传自己的 .so 文件到目标服务器
  2. 能够控制环境变量（putenv 未被禁用）
  3. 存在可触发新进程的 PHP 函数，如 mail()、imap_mail()、mb_send_mail()、error_log() 等

选择劫持哪个函数？ 可通过 readelf -Ws /usr/sbin/sendmail 查看 sendmail 调用的库函数，或通过 cat 1.txt | grep execve 查看进程调用了哪些函数。getuid() 和 geteuid() 是常用选择，因为 sendmail 启动阶段必定调用。

第四章：代码执行

4.1 原理与危害

把用户提交的参数直接作为代码去执行，称为代码注入。当动态代码执行函数的参数过滤不严格时，用户输入的数据会被当作服务端代码执行。

为什么存在代码执行？ 应用有时为了代码的简洁性和灵活性，会在代码中调用 eval、assert 之类的函数来处理动态逻辑。

危害分级：

  1. 执行 PHP 代码：获取服务器内容或相关信息（如 phpinfo()）
  2. 写入 Webshell：利用 PHP 代码功能，向服务器写入一句话木马

当 $pattern 的修饰符中包含 /e 时，$replacement 会被当作 PHP 代码执行（PHP 5.5 已弃用，7.0+ 已移除）。

4.9 漏洞发现与防御

漏洞发现

  1. 代码审计：最主要的方式，借助 RIPS、Seay 等代码审计工具可高效发现
  2. 已知 CMS 漏洞：每年都有大量 CMS 爆出代码执行漏洞，关注 Security Advisory
  3. 页面传参测试：重点关注有传入参数的位置，尝试传入恶意代码片段
  4. 黑盒测试：观察参数是否直接拼接进 eval/assert 等函数

防御方法

方法 说明
避免使用危险函数 使用 JSON 保存配置等替代方案，无需 eval
黑名单 + 白名单 必须使用 eval 时，严格处理用户输入，白名单优先
转义特殊字符 使用单引号包裹可控代码，插入前用 addslashes()、htmlspecialchars()、htmlentities() 等转义
禁用 e 修饰符 放弃 preg_replace 的 /e，改用 preg_replace_callback()
禁用危险函数 在 php.ini 中 disable_functions 配置，禁用 eval、assert 等
