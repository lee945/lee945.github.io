SQL 注入概述

定义

SQL 注入是一种将 SQL 代码插入或添加到应用（用户）的输入参数中，再将这些参数传递给后台 SQL 服务器加以解析并执行。如果 Web 应用未对动态构造的 SQL 语句所使用的参数进行正确性审查，那么攻击者就很可能修改后台 SQL 语句的构造。一旦攻击者能够修改 SQL 语句，该语句将拥有与应用用户相同的运行权限。

Web 页面类型

类型 特点 安全性
静态网页（.html/.htm） 不依赖数据库，灵活性差，交互性差 安全，不存在 SQL 注入
动态网页（.php/.asp 等） 依赖数据库，灵活、交互性强，维护方便 存在 SQL 注入风险
伪静态网页（.html 伪装） 通过 URL 重写隐藏动态参数，本质仍是动态 测试时可直接对 .html 前参数加 ' 判断

动态网站工作流程

SQL 注入起源

  1. 程序员水平和经验欠缺，对 SQL 注入没有防范意识
  2. 许多教程或模板存在 SQL 注入漏洞
  3. 网上给出的许多解决办法并未从根本上解决问题
  4. 数据库管理员对数据库权限设置不合理

SQL 注入典型攻击手段

  1. 判断应用程序是否存在注入漏洞
  2. 收集信息，判断数据库类型
  3. 根据注入参数类型，重构 SQL 语句原貌
  4. 猜解当前库名、表名、字段名，查询数据
  5. 获取账户信息、攻击 Web 或为下一步攻击做准备

SQL 注入危害

  • 数据库信息泄漏
  • 网页篡改
  • 数据库被恶意操作（增删改）
  • 服务器被远程控制
  • 种植木马（写入 webshell）

堆叠查询

原理

PHP 中 mysqli_multi_query() 支持一次执行多条 SQL 语句（用 ; 分隔）。如果后端未过滤 ;，攻击者可在正常语句后追加任意 SQL。

与 UNION 的区别： 堆叠可执行 INSERT / UPDATE / DELETE / DROP 等任意语句，UNION 只能追加 SELECT。

宽字节注入

背景

当某字符大小为 1 字节时称窄字节（英文、数字），2 字节时称宽字节（中文等）。常见宽字节编码：GB2312、GBK、GB18030、BIG5、Shift_JIS。

GBK 编码规则：第一个字节 ascii > 128（即 hex > 0x80）时，与下一字符合并为一个汉字。

原理

PHP 使用 addslashes() 或 magic_quotes_gpc 开启时，’ 被转义为 \’（URL 编码：%5c%27）。

核心：%df 吃掉 %5c，让转义符 \ 失效。

Cookie 注入

Cookie 是什么

Cookie 是在 HTTP 协议下，服务器或脚本可以维护客户工作站上信息的一种方式。通常用于辨别用户身份、进行 session 跟踪，最典型的应用是保存用户账号密码用来自动登录，以及电商网站的”购物车”。

Cookie 注入识别步骤

  1. 寻找形如 .asp?id=xx 的带参数 URL
  2. 去掉参数测试页面是否仍正常（验证参数是否通过 Cookie 传递）
  3. 测试应用是否用 Request("id") 获取数据（同时接受 GET/POST/Cookie）
  4. 将注入语句带入 URL（和常规注入一样），分别返回正常/异常则存在注入
  5. 使用常规注入语句进行注入

二次注入

原理

二次注入分为两步：

第一步：插入恶意数据。 第一次向数据库插入数据时，仅对特殊字符进行了转义，写入数据库时保留了原始恶意数据。

第二步：引用恶意数据。 数据存入数据库后，开发者认为数据是可信的，再次查询时直接从数据库取出恶意数据，没有做进一步检验和处理，导致注入。

关键在于信任边界： 数据入库时做了转义但仍保留了恶意内容，后续使用时未再次过滤。
